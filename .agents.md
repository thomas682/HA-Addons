## API Documentation Policy (MANDATORY)

### Rule: Every new or changed API endpoint MUST be documented immediately and completely.

This rule is mandatory. It applies to every contributor (agents and humans) working on this InfluxBRO codebase.

### When this rule applies

- A NEW endpoint is added (a new route is registered)
- An existing endpoint is CHANGED (request parameters, response shape, errors)
- An endpoint is REMOVED (remove the doc entry or mark it deprecated)
- The URL or HTTP method changes
- Error codes / timeouts / performance characteristics change

### What must be documented

Update the API documentation data structure (the `API_DOCS` list used by the UI) for the endpoint.

Required fields per endpoint:

- id
- method
- path
- summary (one sentence for beginners)
- description (technical explanation for advanced users; may include HTML)
- group
- tags
- role / roleLabel / roleColor
- authRequired
- request: contentType + schema/keys + example
- response: contentType + schema + example + statusCodes
- timing: typicalMs / p95Ms / timeoutMs / bottleneck
- correlations: dependsOn / triggers / parallelWith / usedIn / criticalPath / phase
- errors: symptom / cause / fix (or omit section if none)
- examples: curl / javascript / python (copy-paste ready)
- notes

### Quality criteria

1. No placeholders: no "TODO", "...", "TBD".
2. Examples must use the real server base URL (the running InfluxBRO host/port) and real Home Assistant entity_ids.
3. Summary must be understandable for non-programmers; description must be technically precise.
4. If you change endpoints, update their documentation in the same commit/PR.

### Violation

If an endpoint is merged without complete documentation, it is treated as a bug and must be fixed immediately.
